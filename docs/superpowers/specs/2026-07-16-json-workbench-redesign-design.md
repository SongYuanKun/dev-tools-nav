# JSON 工作台重设计

## 背景

当前规范地址 `/tools/json/` 只是一个通用外壳，通过 iframe 加载 `/pages/tools/json.html?embed=1`。真实工具内部仍包含另一套页面结构，导致嵌套滚动、焦点割裂、可用高度受限和移动端体验不稳定。工具本身同时展示输入和格式化副本，常用操作分散在两行工具栏与折叠面板中；格式化、树形、JSON Path、YAML 和结构对比没有统一的工作流。

现有 `js/json-tool.js` 还把解析、修复、格式化、树形渲染、YAML、Diff、DOM 操作和状态持久化集中在一个 1032 行脚本中。这不仅难以测试，也使任何界面调整都容易破坏已有能力。

## 目标

- `/tools/json/` 成为无需 iframe 的完整规范页面，桌面和移动端都能直接使用。
- 以一个 CodeMirror 6 编辑器作为 JSON 内容的唯一数据源，提供语法高亮、行号、折叠、搜索、撤销、错误标记和定位。
- 将格式化、树形、JSONPath、YAML 和 Diff 组织为清晰的工作模式；只有确实需要两个输入的 Diff 模式使用双编辑器。
- 保留并改善格式化、压缩、复制、清空、上传、下载、示例、修复、Unicode 转换、Key 排序、宽松解析和偏好设置。
- 所有内容继续仅在浏览器本地处理，不上传、不用 `eval`，也不持久化用户输入。
- 保留现有有效使用埋点口径，补齐搜索友好的教程、FAQ 和结构化数据。
- 通过纯逻辑单元测试、浏览器交互测试、可访问性检查、截图和全站回归建立发布门禁。

## 非目标

- 本次不重做其他九个自建工具的通用外壳。
- 不引入账号、云同步、服务端解析、协作编辑或 AI 修复。
- 不支持完整 JSONPath 标准的所有过滤表达式；首版保持当前常用的对象键与数组下标查询，并明确能力边界。
- 不把 YAML 当作独立文件编辑器产品；它是同一数据在 JSON 与 YAML 表示之间的转换模式。
- 不在本任务中完成 Markdown 单一来源流水线；JSON 工作台合入后立即恢复该目标。

## 信息架构与交互

### 页面层级

规范页直接包含站点导航、工作台、使用说明、FAQ、相关工具和页脚。首屏核心结构为：

1. 紧凑标题区：工具名称、用途说明和“本地处理”隐私标识。
2. 模式栏：`格式化`、`树形`、`JSONPath`、`YAML`、`Diff`。
3. 主操作栏：`格式化` 为主要动作，`压缩`、`复制`、`清空` 始终可见。
4. 次要操作区：上传、下载、示例、修复、Unicode 转义/还原、Key 排序和设置。
5. 工作区：模式相关视图。
6. 状态栏：有效性、行列位置、字符数、字节数和当前缩进。

桌面端工作区占据主要视口高度；移动端改为单列，模式栏可横向滚动，常用动作保持可触达。工具不能产生页面内外双重滚动条。

### 单一内容状态

主 JSON 文本是应用状态的唯一来源：

- 格式化模式直接编辑主文本，不再并排显示重复的格式化输出。
- 树形模式解析主文本并生成可折叠、可复制路径的只读树；返回格式化模式时内容不变。
- JSONPath 模式查询主文本，展示匹配值、路径和匹配数量；结果可以复制。
- YAML 模式从主 JSON 生成 YAML，允许编辑 YAML，并通过明确动作写回主 JSON。转换失败时不覆盖任何一侧。
- Diff 模式以主 JSON 为左侧初始值，右侧是独立临时输入；差异结果不覆盖主文本，除非用户明确选择应用某一侧。

模式切换不得隐式格式化、修复或覆盖内容。浏览器只在 `localStorage` 中保存缩进、宽松解析、转义显示和主题等偏好，不保存编辑内容。

### 错误与修复

解析错误使用编辑器诊断标记和工作区顶部错误横幅同时展示。错误信息必须包含可获得的行、列和简短原因；点击横幅或“定位”按钮将光标移动到错误位置并聚焦编辑器。

宽松解析允许注释和尾逗号。单引号等高风险自动转换不在静默解析中发生，只能通过“尝试修复”触发。修复前后内容不同时展示预览或明确确认，失败时保留原文。任何格式化、压缩、排序、YAML 写回操作都采用“先完整校验，后一次性提交”的方式，避免部分覆盖。

### 键盘与可访问性

- 模式栏使用正确的 tablist/tab 语义和方向键导航。
- 所有图标按钮有可读名称，状态和错误通过适度的 `aria-live` 区域播报。
- `Ctrl/Cmd+Enter` 格式化，`Ctrl/Cmd+Shift+M` 压缩，`Ctrl/Cmd+K` 聚焦 JSONPath；快捷键同时显示在帮助中。
- 焦点样式、错误颜色和 Diff 颜色同时依靠文字/符号，不仅依靠颜色传达。
- 对话框、菜单和设置抽屉支持 Escape 关闭，并将焦点还给触发元素。
- 尊重 `prefers-reduced-motion`，触控目标不小于 44 CSS 像素。

## 技术架构

### 页面与兼容地址

- `tools/json/index.html` 改为真实完整页面，承载 canonical、Open Graph、SoftwareApplication、HowTo 与 FAQPage 数据。
- `pages/tools/json.html` 缩减为 `noindex` 的兼容跳转页，保留查询参数并跳转到 `/tools/json/`；删除 `embed=1` 分支。
- 截图脚本直接访问 `/tools/json/`，不得再依赖 iframe 或旧页 DOM。
- 其他工具的通用 shell 本次保持不变。

### 模块边界

- `js/json-core.mjs`：不访问 DOM 的纯函数，包括严格/宽松解析、错误位置、格式化、压缩、递归 Key 排序、统计、JSONPath、Unicode、JSON/YAML 转换和结构 Diff。
- `js/json-workbench.mjs`：应用状态、CodeMirror 扩展、模式控制、命令、文件读写、剪贴板、偏好与埋点。
- `js/json-workbench.bundle.js`：由构建生成并跟踪的浏览器 bundle，HTML 只加载该产物。
- `css/json-workbench.css`：JSON 工作台专属样式。旧 `css/tools.css` 中仅 JSON/iframe 专用规则在确认无引用后删除，通用工具样式保留。

纯逻辑先从旧脚本中提取并用契约测试固定行为，再替换界面。旧 `js/json-tool.js` 在规范页和兼容页都不再引用后删除，避免维护两套实现。

### 编辑器与构建

使用 CodeMirror 6 的模块化包和 Rollup：

- `codemirror` 提供基础编辑器能力。
- `@codemirror/lang-json` 提供 JSON 语言支持。
- `@codemirror/lint` 提供诊断 UI。
- `@codemirror/merge` 用于 Diff 模式的并排差异视图。
- `rollup`、`@rollup/plugin-node-resolve` 和压缩插件生成稳定的浏览器产物。

`npm run build:json` 生成 bundle，`npm run build` 先运行 JSON 构建再生成 Sitemap。CI 和两套部署工作流必须运行同一完整构建。增加构建漂移检查，确保提交的 bundle 与源码、锁文件一致。

主 bundle 需要压缩和 tree-shaking。以构建测试限制其未压缩体积，首版上限设为 750 KiB；如 Merge 扩展使主包超限，则改为动态加载独立 Diff chunk，不能直接提高上限。

## 数据、安全与性能

- 输入、文件、转换结果和 Diff 内容只存在于当前页面内存；关闭或刷新页面后不恢复。
- 不发送内容到分析、日志或网络端点；Umami 只记录 `tool_used` 的工具名和动作名。
- 现有 `format`、`minify`、`repair`、`validate`、`diff` 动作名继续保留，确保运营报表中的“JSON 格式化”有效使用口径不漂移。
- 上传仅接受 JSON/YAML/文本类型，默认最大 5 MiB；超限时清晰提示。下载文件名和 MIME 固定生成，避免使用用户文本构造 HTML。
- 所有树节点、查询结果、错误文本和 Diff 内容通过 DOM 文本节点或编辑器 API 渲染，禁止拼接未经转义的 `innerHTML`。
- 大输入的自动诊断采用防抖；超过 1 MiB 时关闭非必要的即时树形刷新并提示用户手动刷新，避免阻塞主线程。

## SEO 与内容

页面保留规范 URL `https://tools.songyuankun.top/tools/json/`，并提供与真实能力一致的标题、描述和 SoftwareApplication 数据。首屏之后增加：

- JSON 格式化、校验、修复、树形查看、JSONPath、YAML 和 Diff 的简明使用说明。
- “数据是否上传”“宽松解析做了什么”“JSON 与 YAML 转换的限制”“如何定位错误”等 FAQ。
- HowTo 与 FAQPage JSON-LD，内容必须与页面可见文本一致。

不使用关键词堆砌，不宣传未实现的完整 JSONPath 或无损 YAML 能力。

## 测试与发布门禁

### 单元与构建测试

- 严格/宽松解析、字符串内注释符号、尾逗号和错误行列。
- 格式化、压缩、Key 排序、Unicode 往返和原文不被失败操作覆盖。
- JSONPath 的对象、数组、根路径、缺失路径和非法表达式。
- YAML 标量、对象、数组、嵌套、引用字符串和失败保护。
- Diff 的新增、删除、修改与无差异。
- Rollup 产物可复现、体积不超过预算，完整构建无漂移。
- 兼容页指向 canonical，规范页不包含 iframe 和旧脚本。

### 浏览器验收

Playwright 在桌面和移动视口验证：

1. 输入有效 JSON，格式化、压缩、撤销、复制和清空。
2. 输入错误 JSON，显示诊断，点击定位后光标到达目标行列。
3. 五个模式切换时主 JSON 保持不变，树、JSONPath、YAML 与 Diff 的关键流程可用。
4. 上传、下载、设置持久化和内容不持久化符合隐私约束。
5. 键盘操作、tab 语义、焦点返回、可读名称和无横向溢出。
6. `/pages/tools/json.html` 跳转到 `/tools/json/`，规范页不存在 iframe 请求。
7. 截图任务直接在规范页载入示例并生成新截图。

浏览器测试收集 `pageerror`、失败请求和控制台错误，任何一项出现都失败。最终同时运行全量 `npm test`、`npm run audit:tools`、`npm run build` 和截图准备流程。

## 迁移与回滚

实现顺序先建立纯逻辑测试和构建链，再搭建新页面，最后切换 canonical、兼容跳转和截图。整个变更在独立 worktree 分支完成；通过审查和全量验证后，以非强制方式更新 `main`。

生成 bundle、依赖锁文件和页面切换在同一提交序列中保持可构建。若上线后发现阻断性问题，可以整体回退 JSON 工作台提交；旧地址仍会随回退恢复，不涉及数据迁移或服务端状态。

## 验收标准

- 访问 `/tools/json/` 时不存在 iframe，页面首屏直接出现可编辑工作台。
- 格式化模式只有一个主 JSON 编辑器，没有重复输出面板。
- 五种模式及所有批准的常用和高级能力通过自动化浏览器测试。
- 错误可准确定位，失败转换不会覆盖原文，内容不会进入持久化或网络请求。
- 桌面与移动端无嵌套滚动和横向溢出，键盘与辅助技术可完成核心流程。
- 旧地址只做 `noindex` 兼容跳转；截图、SEO、埋点和两套部署均使用规范页与完整构建。
- 全量测试、构建漂移、工具审计和截图流程通过，主 bundle 符合体积预算。

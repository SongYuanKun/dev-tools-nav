# Tools 项目可信基础、增长与商业化设计

> 状态：已批准
>
> 批准日期：2026-07-13
>
> 适用项目：`dev-tools-nav`
>
> 北极星指标：有效工具使用次数

## 1. 目标

将 tools 项目从“功能较多但文档、统计和自动化口径不稳定”的静态工具站，逐步建设为数据可信、持续获得搜索流量、能够长期维护并具备广告与联盟变现能力的开发者工具产品。

总体顺序固定为：

1. 可信基础：修正文档、恢复测试、修复截图自动化、建立分站统计。
2. 有效使用增长：强化核心工具、建设 SEO 内容集群、建立 Markdown 单一内容源。
3. 商业化准备：联盟推荐、赞助合作和低密度展示广告。

当前阶段不通过增加工具数量制造表面增长，也不在流量基础薄弱时直接展示广告。

## 2. 已确认的产品决策

### 2.1 北极星指标

北极星指标为“有效工具使用次数”。只有用户真正执行格式化、转换、生成、验证、复制结果等能获得工具价值的动作，才计入有效使用。页面访问、普通导航点击和分类切换不是有效工具使用。

指标层级如下：

1. 有效工具使用用户数与有效工具使用次数。
2. 搜索曝光、搜索点击、回访和工具间导流。
3. 联盟点击、赞助线索和广告收入。

混合 hostname 的总 PV 不得作为项目主要成功指标。

### 2.2 双站与统计

正式域名 `tools.songyuankun.top` 和 GitHub Pages `songyuankun.github.io/dev-tools-nav/` 均继续提供服务，并继续使用现有 Umami website ID 采集数据。

所有生产运营查询必须：

- 输出或过滤 `hostname`；
- 将 GitHub Pages 的 `/dev-tools-nav` 路径前缀规范化后再跨站汇总；
- 明确统计窗口、查询时间、website ID 和事件口径；
- 能分别展示正式域名、GitHub Pages 和合计数据。

不删除历史 Umami 数据。历史数据能按 hostname 回溯的继续保留；无法可靠拆分的记录明确标注限制。

### 2.3 技术路线

当前继续使用 Vanilla HTML、CSS 和 JavaScript，不进行 Astro + Tailwind 整体迁移。

当原创内容达到 30–50 篇，或手工 HTML 模板维护明显拖慢迭代时，重新评估渐进迁移。迁移评估必须包含 URL 稳定性、SEO、工具交互、双站部署和回滚成本。

### 2.4 内容单一事实源

原创内容以 `content/blog/*.md` 为唯一正文源，由构建脚本生成线上 HTML、博客索引、RSS、sitemap 元数据和结构化数据。CSDN 等外部平台只作为分发、摘要或导流渠道，不反向成为主站正文源。

### 2.5 广告与商业化边界

当前不展示广告。先建立可关闭的商业组件、披露规则和统计能力，达到流量阈值后再逐步启用：

- 月独立有效工具用户达到 1,000：测试联盟链接；
- 月独立有效工具用户达到 5,000：评估单个展示广告位。

核心在线工具首屏始终无广告。禁止弹窗、遮挡、自动播放、诱导点击和虚构用户评价。

### 2.6 激活类内容隔离

KMS、JRebel 等激活类内容保留现有入口，但：

- 不进入首页精选或主动推广；
- 不展示广告、联盟链接或赞助内容；
- 在统计中单独分类；
- 不计入商业化转化指标。

## 3. 文档架构

### 3.1 文件职责

- `README.md`：当前已上线能力、准确数量、公开入口、开发和部署命令。
- `manual.md`：使用与日常维护手册，不保存会快速过期的运营快照。
- `docs/roadmap.md`：唯一产品路线图，记录阶段、状态、指标、准入条件和商业化约束。
- `docs/analytics-insights.md`：带日期、窗口、hostname 和查询依据的运营快照；新快照保留历史对比。
- `docs/umami-integration-spec.md`：当前统计架构、事件定义、查询规则和验证方法。
- `docs/chatdev-p1p2-prompt.md`：历史归档，不作为当前实施依据。
- `content/blog/*.md`：原创文章唯一正文源。

关键文档应包含 `last_verified` 或明确快照日期。无法通过代码、线上数据或官方来源验证的信息必须注明来源和日期。

### 3.2 已确认的客观修订

第一阶段修正文档中的以下事实错误：

- 工具目录为 73 条；自研浏览器工具为 10 款；`online-tools` 数据分类为 11 条，三种口径不得混用。
- 公开工具入口为 `/tools/*/`；`pages/tools/*` 是实现页、嵌入页或兼容跳转路径。
- tools 已接入 Umami，website ID 已存在；旧的“尚未接入、待创建”说明删除或标注为历史。
- 当前事件以实际代码和中文映射为准，不继续使用过时的 `category_switch`。
- GitHub Pages 与正式域名共用 website ID，但运营报告必须按 hostname 区分。
- 项目已有 `scripts/*.test.mjs` 测试；当前基线为 8 项中 7 项通过、1 项失败，不得描述为“无测试”。
- 截图自动化当前不健康，不再宣称“每周自动刷新成功”。
- GitHub Pages、`deploy.sh` 和 1Panel SSH 的排除规则分别记录，不宣称完全一致。
- 已完成的收藏、更新日志、工具互链和 SQL 工具不继续列为未完成需求。
- 品牌色以当前批准的 `#C8956C` 为准；历史 Prompt 中的 `#6366f1` 仅保留在归档上下文。

## 4. 第一阶段：可信基础

### 4.1 测试恢复与 CI 门禁

修复 sitemap 生成器与测试之间的导出契约，使现有测试全部通过。新增独立 CI，在 push 和 pull request 上执行：

```bash
npm ci
npm test
```

CI 不依赖部署工作流间接验证代码。部署前仍可重复运行构建相关检查。

### 4.2 截图自动化

修复 JSON 截图准备流程。脚本必须先打开“更多”菜单，再操作可见的“示例”按钮，或采用同等稳定且能验证真实用户路径的方式。截图脚本需要：

- 对目标可见性给出明确断言；
- 每个截图失败时输出具体页面和步骤；
- 工作流使用可靠清理逻辑停止静态服务器；
- 成功生成首页、博客和 JSON 工具三张有效 PNG；
- 验证 JSON 截图中已加载示例内容。

### 4.3 工具元数据校验

建立自动校验，至少覆盖：

- `TOOLS_DATA` 总数为 73；
- 自研浏览器工具为 10 款；
- 分类数量与唯一 ID；
- canonical 工具路径不重复；
- README 中的数量和公开路径与数据源一致。

工具数量发生有意变化时，应先修改数据，再同步测试预期和文档，不由文档单独定义数量。

### 4.4 分站运营报表

更新运营 SQL 和文档，固定输出：

- hostname；
- 规范化路径；
- PV、会话、访客；
- 有效工具使用次数和用户数；
- 时间窗口与对比窗口。

正式报表禁止只按 website ID 和 url_path 聚合。KMS/JRebel 在查询中单独标记，商业化报表默认排除。

### 4.5 第一阶段验收

- `npm test` 全部通过；
- CI 在 push 和 pull request 上运行测试；
- 三张截图在本地和 GitHub Actions 均稳定生成；
- 正式域名和 GitHub Pages 的统计可分别查看；
- 报表合计值等于两个目标 hostname 的分站值之和；
- README、manual、roadmap、analytics 和 Umami 规范不存在已知事实矛盾。

## 5. 第二阶段：有效使用增长

### 5.1 核心工具范围

第一批增长对象为 JSON、JWT、SQL、正则、Cron 和时间戳。资源不平均分配给全部目录条目。

每个核心工具建设：

- canonical 工具页；
- 问题型教程；
- FAQ；
- 可复制的真实示例；
- 相关工具互链；
- SoftwareApplication、FAQ 或 HowTo 等适用的结构化数据。

### 5.2 内容流水线

主站 Markdown 构建流程生成：

- 文章 HTML；
- 博客列表与分页所需数据；
- RSS 或 Atom Feed；
- sitemap URL 与真实 `lastmod`；
- Open Graph 和结构化数据。

生成产物不得成为人工编辑的第二正文源。

### 5.3 SEO 与复盘

接入 Search Console 数据，至少观察查询词、页面、曝光、点击、CTR 和收录状态。每月联合复盘：

- 独立有效工具用户；
- 有效工具使用次数；
- 搜索点击；
- 30 天回访。

连续 30 天有效使用用户过低的工具停止功能扩张；有曝光但 CTR 低的页面优化标题和摘要；有访问但无有效使用的工具优先检查搜索意图、首屏价值和操作路径。

## 6. 第三阶段：商业化准备

### 6.1 变现顺序

1. 联盟推荐。
2. 工具赞助与专题合作。
3. 低密度展示广告。

联盟内容优先覆盖域名、托管、IDE、API、安全和开发服务等高购买意图场景。

### 6.2 商业组件与统计

商业组件默认关闭，关闭时不产生空白占位。联盟和赞助点击至少携带：

- `partner`；
- `placement`；
- `tool`；
- `category`。

建立联盟披露、赞助标识、合作说明和隐私说明。付费关系不得改变工具评价结论。

商业实验同时评估收入、出站转化、回访率和 Core Web Vitals。收入提升不能以明显损害核心工具体验为代价。

## 7. 实施与交付边界

- 以最新 `origin/main` 为基线，在隔离 worktree 和独立分支中开发。
- 保留原工作区已有的 `sitemap.xml` 修改，不覆盖或混入本项目提交。
- 第一轮只实施可信基础；内容增长和商业化分别使用后续独立计划。
- 行为变化采用测试先行；配置和纯文档修订使用可复现检查验证。
- 未经用户确认，不直接部署、重启服务或推送生产。
- 每轮交付包含变更摘要、验证证据、剩余风险和上线建议。

## 8. 已知基线问题与风险

- 2026-07-13 基线 `npm test` 为 7 通过、1 失败；失败原因是 sitemap 测试导入了实现未导出的函数。
- 2026-07-06 截图工作流因 `#btnSample` 隐藏而超时；此前还存在连续取消记录。
- 历史 Umami 数据混合多个 hostname；能回溯拆分但不能保证所有旧记录完整。
- 双站共用 website ID 便于汇总，也要求所有生产报表严格遵守 hostname 口径。
- 自动化提交可能与其他定时同步发生并发冲突，截图工作流修复时需要验证推送策略。
- KMS/JRebel 内容存在品牌和广告平台合规风险，因此必须持续保持商业隔离。
- 当前保持 Vanilla 能降低近期迁移风险，但内容数量增长后需要重新评估构建架构。

## 9. 非目标

本设计不包含：

- 当前阶段展示广告；
- Astro + Tailwind 整站迁移；
- 用户账号、后端数据库或登录系统；
- 删除任一站点或清洗历史 Umami 数据；
- 第一阶段新增在线工具；
- 虚构推荐语、用户评价或商业数据。
